## 1. Sonnet 5 vs Sonnet 4.6 — 「高くなった」の正体と結論

### 1.1 消費増の印象は事実（3つの要因）

1. **新トークナイザ**: Sonnet 5 は同一テキストを 4.6 比で**約 +30% のトークン**として数える（Anthropic 公式のマイグレーションガイド記載値）。`/cost` や Usage ダッシュボードの数字が増えて見えるのはまずこれ。
2. **adaptive thinking がデフォルト ON**: 4.6 では thinking 無しだった軽いターンでも思考トークンが出力課金される。
3. **より agentic な挙動**: ツール呼び出し・自己検証を自発的に行うため、1タスクのターン数が伸びやすい。

### 1.2 それでも今は 4.6 に戻すべきでない

| 期間 | Sonnet 5 単価（入力/出力 per 1M） | 等価テキストの実効コスト（4.6 比） |
|---|---|---|
| 〜2026-08-31（イントロ価格） | $2 / $10 | 単価 0.67× × トークン 1.3× ≈ **0.87×（約13%安い）** |
| 2026-09-01 以降 | $3 / $15（4.6 と同額） | **約 1.3×** |

つまり**現時点では「高くなった」は錯覚に近い**（トークン数は増えたが請求額は下がる方向）。9月以降は等価テキストで約1.3倍になるが、その時の答えも 4.6 併用ではない:

- **単純修正の正しい逃し先は Haiku 4.5（$1/$5）**。旧トークナイザのため実効で Sonnet 5 の約 1/4。今回整備するスキル群（§4）でここに流す。
- 4.6 併用を推奨しない理由: (a) プロンプトキャッシュはモデル別で、セッション途中・セッション間のモデル切替はキャッシュ再構築コストを生む、(b) Sonnet 2系統の運用ルール維持コスト、(c) 4.6 は前世代でありいずれ deprecated になる。
- 品質面では Sonnet 5 はコーディング・agentic タスクで前 Opus 級。**手戻りループの削減効果**（コスト構造の支配項。[エグゼクティブAIレポート](./エグゼクティブAIレポート.md) §2 参照）は 4.6 より大きい。

**アクション:** 2026-09-01（イントロ価格終了）をチームカレンダーに登録し、Sonnet 5 の実測コストを再評価する。それまでは既定 `sonnet` を維持。

## 2. Figma MCP・画像コストとプロンプトキャッシュ

### 2.1 まず前提: キャッシュは既に効いている

Claude Code は会話プレフィックス（CLAUDE.md・ツール定義・過去ターンの MCP レスポンスやスクリーンショットを含む）を**自動でプロンプトキャッシュに載せている**。読み取りは入力単価の約 0.1 倍、書き込みは約 1.25 倍、TTL は 5 分。「キャッシュを有効にする設定手順」は存在しない — **すでに ON であり、できるのは「壊さない」ことと「載せる量を減らす」こと**。

Figma のスクリーンショットや `get_design_context` の巨大レスポンスも、取り込んだ後のターンではキャッシュ読みになっている。高いのは「**初回の取り込み**」と「**TTL 切れ後の再送**」。

### 2.2 推奨方針（効果順）

1. **取り込む量の削減が本丸**（キャッシュより効く）。既存ポリシーの徹底で足りる:
   - metadata-first・フルコンテキスト一括取得の禁止・取得済み情報の specs への永続化（= Figma レスポンスの**恒久キャッシュ**として specs を使う）
   - スクリーンショットの枚数・`maxDimension` の抑制、検証は `mode: report` 標準（反復ループを1回で打ち切る）
   - 詳細ルール: [ai-cost-optimization.md §2.2, §6, §7](../ai-cost-optimization.md)
2. **TTL 5分を切らさない**。Figma 作業セッションを放置したり長いレビュー待ちを挟むと、次ターンで全履歴（画像込み）がキャッシュ書き込みとして再課金される。重い画像を積んだセッションほど「集中バーストで進めて区切りで閉じる」が効く。中断が長引くなら `/handoff` → `/clear` で畳む方が安い。
3. **セッション途中で `/model` を切り替えない**（キャッシュはモデル別）。モデル変更はタスク単位・`/clear` 後に行う。
4. **CLAUDE.md・エージェント定義の編集はまとめて行う**。プレフィックスが変わった日は全員の全セッションでキャッシュ書き込みが再発生する。

### 2.3 誤解しやすい点

- 「画像をキャッシュすればタダになる」ではない。**1回目は常にフル課金＋書き込み 1.25 倍**。同じスクショを2度撮らない・不要な解像度で撮らないことの方が効果が大きい。
- サブエージェントはコールドスタート（親のキャッシュを引き継がない）。Figma を読むエージェントには**まとまった単位（コンポーネント1件・画面1枚）を渡す**現行設計を崩さないこと。

## 3. モデル4段戦略とリポジトリ設定の対応

判定ロジック（単価ではなく「単価 × トークン量 × 試行回数」で選ぶ）とエスカレーション基準は [ai-cost-optimization.md §3](../ai-cost-optimization.md) が権威。ここでは**設定箇所との対応**だけまとめる。

| 層 | 単価（入/出 per 1M） | 用途 | リポジトリでの設定箇所 |
|---|---|---|---|
| **Haiku 4.5** | $1 / $5 | コミット・PR作成・typo/lint 修正などの定型作業 | `committer`・`pr-creator` エージェントと `/fix-typos`・`/lint-fix` コマンドの frontmatter `model: haiku` |
| **Sonnet 5（既定）** | $3 / $15（イントロ $2/$10） | 実装・抽出・レビュー全般 | `.claude/settings.json` の `model`、既存 agents の `model: sonnet` |
| **Opus 4.8** | $5 / $25 | 複雑な設計判断・収束しない実装（§3.3 チェックリスト該当時） | 設定固定しない。タスク単位で `/clear` → `/model opus` |
| **Fable 5** | $10 / $50 | Opus でも足りない長期自律・高難度タスク（§3.4 該当時のみ） | 同上。**日常セッションを Fable のまま始めない**（thinking 常時 ON で軽いターンにも思考課金が乗る） |

運用上の注意:

- `/model` でモデルを上げた後、**タスク終了時に必ず `sonnet` に戻す**。「Opus/Fable のまま次の軽い作業を始める」が最も典型的な無駄。
- サブエージェントの frontmatter `model:` は「その工程の知的要求」で固定する。メインセッションのモデルに引きずられない（`inherit` にしない）。
- 定型作業をメインセッションのコマンドでなく**サブエージェント委譲**にするのは、メインの会話履歴・キャッシュに触れず、小さなコールドスタート（diff だけ読む）で済むから。コマンドの `model:` 切替はセッション履歴全体を別モデルで読み直すため、長いセッションでは逆に高くつく。

## 4. チームスキル整備（今回導入分）

### 4.1 導入するスキル・エージェント

| スキル | 実行モデル | 方式 | 役割 |
|---|---|---|---|
| `/commit` | Haiku | `committer` サブエージェント委譲 | 作業の区切りでの日本語コミット作成。push はしない |
| `/create-pr` | Haiku | `pr-creator` サブエージェント委譲 | `gh pr create`（日本語タイトル・本文）。merge はしない |
| `/fix-typos` | Haiku | コマンド（既存を Haiku 化） | cspell トリアージ。独立タスクとして実行する前提 |
| `/lint-fix` | Haiku | コマンド（新規） | `pnpm run lint` の機械的修正 |
| `/handoff` | セッション既定 | コマンド（新規） | セッション終了時に引き継ぎメモを書き出す → `/clear` 後の再開を安く正確に。「1機能=1セッション」運用の要 |
| `/update-spec` | セッション既定 | コマンド（新規） | 実装完了後に specs/（Source of Truth）とのドリフトを検知し spec を更新。仕様精度は verify 往復数に直結する最重要投資 |

**`/fix-typos`・`/lint-fix` の注意:** コマンドレベルのモデル切替は、実行中そのセッションの履歴を Haiku で読み直す。**タスクの切れ目や独立セッションで実行する**こと（長い実装セッションの途中に挟まない）。

### 4.2 見送ったもの（と理由）

- **`/scaffold-component`**（Atomic Design 雛形生成）: `figma-component-library` フローと役割が重複。Figma を経由しない手作りコンポーネントの需要が実際に出たら再検討。
- **Agent teams / Dynamic workflows**: 評価済み・不採用（[ai-cost-optimization.md §8](../ai-cost-optimization.md) 参照）。

## 5. コミットタイミング方針の変更 — 分析と決定

### 5.1 「作業途中コミットはトークン節約になるか」への正直な回答

**コミット行為そのものは節約にならない。** `git status` / `git diff` / `git commit` で毎回数千トークンの**純増**であり、会話履歴も短くならない。「節約量が大きいなら変更」という基準を文字どおり適用するなら、変更しない判断もありえた。

しかし節約は**間接経路**で発生する:

- 最大の削減施策は「タスクの切れ目で `/clear`」（肥大した履歴を次タスクの全ターンで払い続けない）。**未コミットの作業が積み上がっているとセッションを畳む判断ができず**、この施策が実行されない。こまめなコミットは `/clear` と auto-compact を「いつでも安全」にする基盤になる。
- compaction・セッション再開時、AI は会話の記憶でなく `git log` / `git diff` から状態を復元できる（読み直しが安く正確になる）。

### 5.2 決定

**採用する。** 理由はトークン節約（間接的・中程度）よりも、(a) セッション分割の基盤、(b) コミット差分単位の人間レビュー、(c) ロールバック・引き継ぎの容易さ、の運用価値。

新ルール（CLAUDE.md のコミット規約に反映済み）:

1. AI は**作業の意味的な区切りごとに**フィーチャーブランチへコミットしてよい（人間の指示を待たない）
2. 人間は**コミット差分単位で**変更内容をレビューする
3. **push は人間のみ**。`git push` は `.claude/settings.json` の `permissions.deny` で技術的にブロック（値の権威は同ファイル）
4. `main` への直接コミット禁止・日本語コミットメッセージは従来どおり
