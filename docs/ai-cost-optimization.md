# AI 設定・運用コスト最適化ガイドライン

Figma MCP × Claude Code によるプロトタイプ自動生成ワークフローの運用コスト（Claude Enterprise 従量課金）を最適化するためのチームガイドライン。

- 対象: 本リポジトリの Figma 連携ワークフロー（`figma-component-library` → `figma-screen-flow` → `figma-implement-screen` → `figma-verify-screen`）を運用する全メンバー
- 前提: Claude Code + サブエージェント（`.claude/agents/`）+ ループエンジニアリング（`figma-verify-screen` の修正反復）
- 更新日: 2026-07-10（価格・環境変数は変わりうるため、四半期ごとに公式ドキュメントと突き合わせること）
- 意思決定の分析・背景（Sonnet 5 vs 4.6、キャッシュの実態、コミット方針変更）は [docs/report/リードエンジニア向けAI戦略レポート.md](report/リードエンジニア向けAI戦略レポート.md) を参照

---

## 0. TL;DR（最初にこれだけ読む）

1. **単価が安いモデルに寄せても総額は下がるとは限らない。** 現行世代の単価差は Opus:Sonnet = **約 1.67 倍**。安いモデルでの試行錯誤・レビュー往復が 1.7 倍を超えた時点で総額は逆転する。
2. **モデルは「単価」ではなく「1タスクあたり総トークン × 成功率」で選ぶ。** **メインループの既定は Sonnet**（`settings.json` の `model`）。判断の質が下流の試行回数を大きく左右する場面（複雑な設計判断・収束しない実装）だけ `/model opus` で明示的にエスカレーションする（§3.1〜3.3）。定型的な抽出・実装はSonnet、機械的な整形・分類は Haiku。
3. **Fable 5 は単価 2 倍（Opus 比）。** Opus での試行回数・往復が 2 倍を超える長期・複雑タスク専用の、さらに一段上のエスカレーション先（§3.4）。日常のワークフローは「Sonnet 既定 → 必要時 Opus → それでも足りなければ Fable」という 3 段構成。
4. **最大のコスト削減はモデル選定ではなく「往復回数の削減」と「キャッシュヒット率」。** 仕様（specs/）の精度を上げて verify の指摘を 1 round 目で出し切ることが、どのモデル変更よりも効く。
5. 停止条件（maxRounds・同一エラー回数・差分なし検知）を必ず入れる。無限ループ 1 回の暴走は数日分の通常利用に相当しうる。
6. **検証の標準は人間チェック（`figma-verify-screen` の `mode: report`）。** AI は round 1 の指摘レポートまで、修正判断と細部検証は人間が担う。「report + 人間チェック + 実値つき一括修正指示」が標準運用（§7）。自動修正ループ（`mode: loop`）は例外で、ユーザーが明示的に指定した場合のみ使う。

## 1. 「Sonnet 化でコスト増」の原因分析と検証方法

### 1.1 単価の事実確認（2026-06 時点・API 換算 / 1M トークンあたり）

| モデル | 入力 | 出力 | Opus 4.8 比 |
|---|---|---|---|
| Claude Fable 5 | $10.00 | $50.00 | **2.0x** |
| Claude Opus 4.8 | $5.00 | $25.00 | 1.0x |
| Claude Sonnet 5 | $3.00（〜2026-08-31 は $2.00） | $15.00（同 $10.00） | 0.6x |
| Claude Sonnet 4.6 | $3.00 | $15.00 | 0.6x |
| Claude Haiku 4.5 | $1.00 | $5.00 | 0.2x |

ポイント:

- **Opus と Sonnet の単価差は現行世代で約 1.67 倍**。単価だけでモデルを選ぶと、往復回数の差で簡単に逆転する（§1.2）。
- プロンプトキャッシュの読み取りは **入力単価の約 0.1 倍**、書き込みは約 1.25 倍（5分TTL）。キャッシュヒット率が総額を大きく左右する。
- **Fable 5 は単価だけでなく課金の構造が Opus と違う**。使い分けは §3.4 参照:
  - **thinking 常時 ON（無効化不可）**。軽い対話ターンでも思考トークンが必ず出力課金される。短い定型タスクには構造的に不利。
  - 1 ターンが長い（数分〜十数分の自律実行が正常動作）。「1 回のターンで完了する仕事量」が Opus より大きい前提のモデル。
  - 30 日データ保持が必須（ZDR 設定の組織では全リクエストが 400 エラー）。**本番プロジェクトへ設定をコピーする際は組織のデータ保持設定を確認すること**。

### 1.2 なぜ Sonnet 化で総額が上がりうるか

コストは `単価 × トークン量 × 試行回数` の積。Sonnet 化で下がるのは単価（0.6x）だけで、以下が増えると簡単に相殺・逆転する:

| 増加要因 | 本ワークフローでの具体例 |
|---|---|
| **レビュー往復の増加** | `figma-verify-screen` の修正 round が 1→3 に増えると、round ごとにスクショ（画像トークンは重い）+ ソース読み込み + 修正の全コストが再発生 |
| **実装ミスのリトライ** | lint エラー・トークン対応表の読み違い → 再実装。仕様の読み落としが Opus より起きやすい |
| **判断ミスによる手戻り** | メインループ（オーケストレーター）の指示が曖昧 → サブエージェントが誤った成果物を返す → 再スポーン（コールドスタートでキャッシュゼロから） |
| **出力の冗長化** | 同じ成果物を作るのに探索・narration が増える |

**目安: Sonnet で総トークン量（≒試行回数）が Opus 比 1.7 倍を超えるタスクは、Opus の方が安い。**

### 1.3 検証方法（チームで実施すること）

原因が「使用量増」か「モデル特性」かは、次の A/B 計測で切り分ける:

1. **同一画面を 2 通りで実装する**（条件を揃える: 同じ Figma ノード、同じ specs 整備状態、`/clear` 済みの新規セッション）
   - パターン A: メインループ = Opus、サブエージェント = Sonnet（§3.3 の基準に該当するタスクでの明示的エスカレーション）
   - パターン B: すべて Sonnet（`settings.json` の既定構成。通常タスクはこちらが基準）
2. **各セッションで記録するメトリクス**:
   - セッション終了時の `/cost`（Claude Code のセッション内コスト表示）
   - `figma-verify-screen` の収束 round 数
   - lint 失敗・再実装の回数
   - Enterprise Console の Usage ダッシュボードでモデル別トークン内訳（cache read / cache write / uncached input / output）
3. **判定**: 「1タスクあたり総コスト」と「成果物の手直し工数」で比較する。B の総コスト ≧ A なら、単価差より試行回数差が支配的という結論になる。

継続的な計測には Claude Code の **OpenTelemetry メトリクスエクスポート**（`CLAUDE_CODE_ENABLE_TELEMETRY=1` + OTEL 設定）を有効化し、モデル別・ユーザー別のトークン消費をダッシュボード化するのが望ましい（Enterprise 管理者向け。設定方法は Claude Code 公式ドキュメント「Monitoring / OpenTelemetry」参照）。

## 2. ① コスト低減のための設定・運用施策

### 2.1 設定ファイル（`.claude/settings.json`）

プロジェクト共有設定はリポジトリの [`.claude/settings.json`](../.claude/settings.json) に置く（チーム全員に適用。agents/ ・ skills/ と同じ階層で自動読込）。個人上書きは `.claude/settings.local.json`。

**設定の権威は同ファイルであり、本節はそこに書かれた方針の説明に留める**（値の二重管理を避けるため、現行の設定内容は必ず実ファイルを参照すること）。方針の要点:

- **メインループの既定モデルは `sonnet`**。判断の質が問われる複雑タスクは `/model opus` で明示的に切り替える（§3.1〜3.3）。
- 個別セッションで出力上限等の調整が必要になった場合は `.claude/settings.local.json` で上書きする。`env` を設定する場合、環境変数名はバージョンで変わることがあるため `claude` の公式 Settings ドキュメントで現行名を確認すること。

### 2.2 コンテキスト肥大化を防ぐ運用ルール

コンテキストが太る = 毎ターンの入力トークンが太る（キャッシュがあっても書き込み分は課金される）。

| ルール | 理由 |
|---|---|
| **タスクの切れ目で `/clear`** する（画面 1 枚の実装完了ごと等） | 前タスクの残骸コンテキストを次タスクの全ターンで払い続けない |
| 長丁場では区切りで `/compact` を明示実行 | 自動コンパクトを待つより制御しやすい |
| **CLAUDE.md は薄く保つ**（現状維持。増やすなら参照リンク方式） | CLAUDE.md は全セッション・全ターンの固定費 |
| サブエージェントの返答は**要約 JSON のみ**（生の `get_design_context` を返させない） | 既存エージェント定義で徹底済み。崩さないこと |
| 同一 Figma ノードの再取得禁止（specs に記録済みの node-id を参照） | 既存スキルで徹底済み。新規スキル追加時も踏襲する |
| **大きなノードは metadata-first**: `get_metadata`（sparse XML）で構造を把握してから、`get_design_context` は必要なセクションのノードに絞って呼ぶ。既実装の共通シェル（Header / Side_Bar 等）や specs 済みコンポーネントのサブツリーは取得しない | design_context のレスポンスは対象サブツリー全体のコード表現。画面ごとに共通シェルぶんを取り直すのが最大の浪費。Figma 公式も大規模デザインでの metadata 先行を推奨している |
| スクショは `maxDimension` を絞る（コンポーネント差分確認は 600〜1000） | 画像はテキストよりトークン単価が実質的に重い |
| 大きなファイルは必要範囲だけ Read（offset/limit） | 全文 Read の積み重ねが効く |

### 2.3 プロンプトキャッシュを壊さない

Claude Code は会話プレフィックスを自動キャッシュする（読み取り 0.1x）。壊す行為を避ける:

- **セッション途中でモデルを切り替えない**（キャッシュはモデル別。切り替えた瞬間に全履歴が非キャッシュで再課金される）。モデルを変えたいなら `/clear` してから。
- CLAUDE.md・エージェント定義の頻繁な編集を避ける（編集当日は全セッションでキャッシュ書き込みが再発生する。まとめて更新する）。
- サブエージェント起動はコールドスタート（親のキャッシュを引き継がない）。**細かいタスクで多数スポーンするより、1 スポーンにまとまった単位（コンポーネント 1 件・画面 1 枚）を渡す**。現行設計の粒度が適正。

## 3. ② モデルの使い分け（ベストプラクティス）

### 3.1 選定ロジック

判断基準は単価ではなく **「期待総コスト = 単価 × (1タスクの平均トークン量) × (平均試行回数)」**。

- **判断の質が下流の試行回数を左右する工程 → 既定は Sonnet、複雑な設計判断・曖昧な仕様解釈が絡むタスクだけ `/model opus` へ明示的に切り替える**（§3.3）。オーケストレーター（メインループ）の指示ミスはサブエージェント丸ごと 1 回分の無駄になるため、切り替え判断自体はケチらない。
- **入出力の契約が明確で、1 回で終わる定型工程 → Sonnet。** 仕様が固まっていれば実力差が出にくい。
- **知的判断がほぼ不要な工程 → Haiku。** 分類・整形・単純チェック。

### 3.2 本ワークフローへの当てはめ（推奨構成）

| 工程 | 担当 | 推奨モデル | 理由 |
|---|---|---|---|
| メインループ（オーケストレーション・仕様判断・トークン統合） | メインセッション | **Sonnet（既定）**。判断の質が問われる複雑タスクは `/model opus` へ明示切替 | 指示の質が全サブエージェントの成否とループ回数を決める。定型的な指示で足りるタスクはSonnetで十分（§3.1, §3.3） |
| コンポーネント抽出（`figma-component-extractor`） | サブエージェント | **Sonnet**（現状維持） | 入出力契約が明確・1 件 1 回で完結 |
| 遷移図マッピング（`figma-screen-flow-mapper`） | サブエージェント | **Sonnet**（現状維持） | 同上 |
| 画面実装（`figma-screen-implementer`） | サブエージェント | **Sonnet**。ただし **verify で 3 round 収束しない画面が続くなら Opus を試す** | 複雑な画面では Opus の 1 発実装の方が「Sonnet + 修正往復」より安いケースがある（§1.2） |
| 視覚 QA（`figma-screen-reviewer`） | サブエージェント | **Sonnet**（現状維持） | 指摘の見落としは次 round で拾える。Read-only で安価 |
| 定型作業: 自律の区切りコミット（`committer`）・PR 作成（`/create-pr` → `pr-creator`）・typo 修正（`/fix-typos`）・lint 修正（`/lint-fix`） | サブエージェント / コマンド | **Haiku（設定済み）** | 知的判断が薄い。作業フロー内のコミット・PR はサブエージェント委譲にすることでメインセッションの履歴・キャッシュに触れない |
| ユーザーの明示依頼によるコミット（`/commit` → commit スキル） | メインセッション直接 | メインのモデルのまま | メインは作業文脈を既に持っており、エージェント起動して文脈を再構築させるより速くて確実。手順の権威は `.claude/skills/commit/SKILL.md` |

> エージェント定義の frontmatter `model:` で個別指定できる（現状すべて `sonnet`）。メインループのモデルは `settings.json` の `model`（既定 `sonnet`）で指定し、判断の質が問われるタスクは `/model opus` で明示的にエスカレーションする（§3.3）。Opus でも足りない例外的なタスクのみ、§3.4 の基準で Fable 5 へさらにエスカレーションする。

### 3.3 「Opus の方が安い」判定チェックリスト

以下に 2 つ以上当てはまるタスクは Opus に切り替えて A/B する:

- [ ] verify ループが定常的に 3 round 使い切っている
- [ ] 同じ種類の実装ミス（トークン対応表の読み違い・Auto Layout の誤解釈）が繰り返し発生する
- [ ] サブエージェントの再スポーン（やり直し）が発生している
- [ ] タスクが仕様化しきれない曖昧さを含む（新規パターンの画面・複雑な状態遷移）

### 3.4 Fable 5 を使うべき状況

判断基準は §1.2 の「Sonnet 1.7 倍ルール」と同型: **Fable:Opus = 2.0x なので、「Opus での総トークン量（試行回数 × 往復）が Fable の 2 倍を超える」タスクだけ Fable が安い**。単価が倍でも、1 発で終わるなら「Opus × 複数往復 + 手戻り」より総額が下がる — これが成立する範囲だけで使う。

**Fable が有利になりうるタスク**（コスト面でも勝ちうる）:

- **Opus でも収束しない複雑画面の一発実装**: verify が maxRounds を使い切る・再実装が繰り返される画面。Fable の 1 発実装 < Opus × 3 round + 再スポーンなら逆転する。
- **長時間の自律実行**（複数画面の一括実装・夜間ラン）: 途中の手戻りや人間の介入 1 回のコスト（待ち時間・再指示・コンテキスト再構築）が、単価差より高くつく場合。
- **仕様化しきれない曖昧な設計判断・大規模リファクタリング計画**: 判断ミスが下流の全工程に増幅される工程。§3.1 の「判断の質が問われる場面は Opus へ昇格」の考え方の、さらに一段上のエスカレーション版。

**Fable を使わない状況**（Opus / Sonnet のまま）:

- **サブエージェント全般**: 入出力契約が明確な定型工程に Fable 級の判断力は不要。frontmatter の `model: sonnet` は維持。
- **対話的な軽いタスク・質問応答**: thinking 常時 ON が純粋なオーバーヘッドになる（§1.1 の注記参照）。
- **verify のレビュアー**: Read-only の視覚 QA。指摘の見落としは人間チェックで拾える。

**運用ルール**:

- `settings.json` のピンは `sonnet` のまま（プロジェクト標準）。Opus への昇格も Fable への昇格も**タスク単位で `/model` で切り替える**運用に統一する。
- **切り替えは必ず `/clear` してから**。キャッシュはモデル別（§2.3）なので、セッション途中の切り替えは全履歴が非キャッシュで再課金される。
- タスク終了後は `/model sonnet` に戻す（Opus/Fable のまま次の軽いタスクを始めるのが最も典型的な無駄）。

**Fable エスカレーション判定**（§3.3 と同じく、2 つ以上該当したら Fable で A/B する）:

- [ ] §3.3 の基準で Opus に切り替えたが、それでも改善しなかった
- [ ] 1 タスクの想定が Opus で 5 往復以上（指示 → 修正指示の繰り返しが前提になっている）
- [ ] 数時間規模の自律実行で、途中の人間の介入をゼロにしたい
- [ ] タスクの完了条件は明確だが、道筋の設計自体が難しい（実装方針・アーキテクチャの判断が作業の中心）

## 4. ③ 安全性の確保と無限ループ防止

### 4.1 停止条件の設計（ループエンジニアリングの必須要件)

すべての自律ループには**複数の独立した停止条件**を入れる。1 つでは足りない。

| 停止条件 | 実装場所 | 現状 / 推奨値 |
|---|---|---|
| **Max iterations** | スキル・エージェント定義に明記 | `figma-verify-screen` は `maxRounds` 既定 3（維持）。新規ループ系スキルにも必ず同等の上限を書く |
| **同一エラー 2 回で停止** | スキルの手順に明記 | 同じ指摘・同じ lint エラーが 2 round 連続したら、修正を続けず**ユーザーに報告して停止**（原因が仕様側にある兆候） |
| **差分なし検知** | 同上 | 修正 round で実装差分が発生しなかったら即停止（空回りの検知） |
| **サブエージェント側のツール回数上限** | エージェント定義 | 「`get_design_context` は 1 回だけ」等を明記（既存定義で実施済み。新規エージェントでも踏襲） |
| **コスト上限（組織レベル）** | Enterprise Console | 管理者がワークスペースの **spend limit / usage alert** を設定する。個人の運用ミスの最終防波堤 |
| **セッション内の目視確認** | 運用ルール | 長時間の自律実行を開始する前後で `/cost` を確認し、1 タスクの想定コストレンジ（チームで基準値を決める）を超えたら中断して原因を見る |

### 4.2 レビューゲートの設定基準

自律ループに「修正させ続ける」のではなく、ゲートで人間に返す基準を決めておく:

1. **severity の構造化**: レビュアーの指摘を critical / major / minor に分類し、**自動修正の対象は critical / major のみ**。minor は報告に留めて人間が判断（minor の追い込みが round を浪費する典型パターン）。
2. **PASS 基準の明文化**: 「critical/major ゼロ = PASS」。「完璧な一致」を目指させない（ビジョンの限界により、数 px 差の追求は幻覚と往復を生む — `figma-screen-reviewer` の設計思想どおり）。
3. **権限の分離を維持**: レビュアーは Read-only（修正権限なし）、実装エージェントは検証権限なし。既存設計を崩さない。相互に役割を侵食させないことが暴走防止になる。
4. **maxRounds 到達時は必ず人間へ**: 残指摘の一覧と「仕様側の問題の可能性」を添えて報告し、続行判断はユーザーが行う。エージェント自身に round 延長を判断させない。
5. **破壊的操作はループの外**: git push・ファイル削除・`@theme` 共有ファイル編集などはサブエージェントに許可しない（既存の「@theme はメインループに一本化」ルールを維持）。`gh` の deny リストも同様。

## 5. ④ プロンプトの最適化

### 5.1 サブエージェント指示テンプレート

既存エージェント定義（`figma-component-extractor` 等）が良い雛形になっている。新規エージェントは以下の 5 要素を必ず備えること:

```markdown
# 役割
（1 文。「〜1 件を〜するエージェント」と処理単位を明示）

# 入力（呼び出し元から渡される）
（キー名と例。呼び出し元が探索せず渡せる形に）

# 手順（コスト配慮：ツール呼び出しは最小限に）
（各ツールの呼び出し回数上限を明記。「1 回だけ」「必要な場合のみ」）

# 制約
（書き込み禁止ファイル・使用禁止ツール・越境禁止の明示）

# 出力フォーマット（厳守・これ以外を出力しない）
（コンパクトな JSON スキーマ。生レスポンスの転送禁止）
```

効果の内訳: 入力契約 → 探索トークンの削減 / ツール回数上限 → 読み取りコストの上限化 / 出力契約 → メインループに戻るトークンの最小化。

### 5.2 ユーザー → メインループの指示の出し方

- **最初のプロンプトに全仕様を入れる。** 対象 URL（node-id 付き）・対象プロトタイプ・完了条件を最初に渡す。小出しにすると各ターンで全履歴を再送するため、後出しの 1 情報が高くつく。
- **完了条件を先に言う**（例: 「verify が PASS したら終了。maxRounds 超過時は報告のみ」）。エージェントが「気を利かせて」追加作業する余地を消す。
- **やらないことを言う**（例: 「リファクタリング不要」「既存画面には触らない」）。
- 質問・調査だけのときは「実装はまだしない」と明示する。

### 5.3 トークンを浪費する指示のアンチパターン

| アンチパターン | 代替 |
|---|---|
| 「全部の画面をチェックして」 | 対象画面を列挙して 1 画面ずつ（並列サブエージェント化も可） |
| 生ログ・生レスポンスを会話に貼る | ファイルに保存してパスを渡す |
| 「もっと Figma に近づけて」（無限の曖昧指示） | 差異を specific に指摘（「ヘッダーの gap が仕様の 16px でない」） |
| セッションをまたいで文脈を口頭で再説明 | specs/ に書いてあることは「specs の◯◯参照」で済ませる |

## 6. ⑤ その他の施策

1. **仕様（specs/）への投資が最大のコスト削減。** Component Library 仕様の「Anatomy / Layout（Fill/Hug/Fixed・gap・padding 実数値）」「Tokens 対応表」の精度が verify の round 数を直接決める。仕様が曖昧なまま実装フェーズに進むのが最も高くつく。抽出フェーズで仕様の抜けに気づいたら、実装前に直す。
2. **Figma MCP レスポンスの「キャッシュ」= specs。** 一度取得した node-id・トークン定義・構造情報は specs に永続化し、二度と MCP から取らない（既存ルール。徹底度を PR レビューで確認する）。
3. **独立タスクは並列サブエージェントで。** コンポーネント N 件の抽出はコンテキストを共有しないため、直列で 1 セッションに積むより、並列スポーンの方が総トークンが小さい（メインループの履歴が太らない）。
4. **計測なき最適化をしない。** Enterprise Console の Usage ダッシュボード（モデル別・メンバー別）を週次で確認する係を決める。§1.3 の OTEL エクスポートを導入すれば自動化できる。
5. **Sonnet 5 のイントロ価格（〜2026-08-31: $2/$10）** の期間中は Sonnet 系タスクを Sonnet 5 に寄せると 33% 安い。期限後に単価が戻ることをカレンダーに入れておく。
6. **モデル・価格情報の定期見直し。** モデル世代交代で単価と性能バランスは変わる。四半期ごとに本ドキュメント §1.1 を更新する。Figma MCP のツール構成も接続形態で可用性が変わる（`search_design_system` / `download_assets` 等は remote 接続限定）ため、同じタイミングで実ツール一覧（`claude mcp list` と当該サーバーのツール一覧）を確認する。
7. **Code Connect は保留（将来の選択肢）。** Figma の Code Connect（ノード⇔コードの対応登録）を導入すると、`get_design_context` の応答が生マークアップでなく既存コンポーネント参照になり、応答トークンの削減と実装の再利用性向上が見込める。ただし登録は Figma への書き込み（本リポジトリの「Figma MCP は read 系のみ」規約の例外）であり、Figma の対応プランも必要なため導入しない。画面実装フェーズで metadata-first（§2.2）を徹底してもなお design_context レスポンスの肥大が実測で問題になったら再検討する。

## 7. 検証の標準は人間チェック（`mode: report`）

`figma-verify-screen` の**標準モードは `report`（人間チェック運用）**。AI は round 1 の指摘レポートを出すまでを担い、Figma との精密な突き合わせと修正指示の作成は人間が行う。自動修正ループ（`mode: loop`）は例外扱いで、**ユーザーが明示的に `mode: loop` を指定した場合のみ**使う。

**スクショ検証ループは画面あたり最もコストの重い工程**であり、この標準化が総額に直結する。

### 7.1 標準（report）と例外（loop）

| | `mode: report`（**標準**・人間チェック運用） | `mode: loop`（**例外**・自動修正ループ） |
|---|---|---|
| 動き | round 1 のレビューのみ。修正せず指摘レポートを出して終了 | PASS まで「スクショ → レビュー → 修正」を反復（maxRounds 上限） |
| AI コスト構造 | 画像 2 枚 + レビュアー起動 × **1 回で固定** | round ごとに画像 2 枚 + レビュアー起動 + 修正が再発生 |
| 人間の役割 | Figma との精密突き合わせ + 修正指示の作成 | 最終確認のみ |
| 使う条件 | **既定。指定不要** | ユーザーの明示指定がある場合のみ（§7.3） |

`report` を標準とする根拠:

- verify ループの反復コスト（画像トークン + レビュアーのスポーン + 修正 + 再スクショ）が 1 回で打ち切られる。
- 人間の指摘はピンポイントなので、修正 1 件あたりは探索なしの小さな Edit で済む。
- 人間は、レビュアーエージェントが構造的に苦手な領域（数 px の余白差・微妙な色差 — `figma-screen-reviewer` 自身が「画像で目測しない」と定めている領域）こそ得意。**品質面でも合理的な分担**になる。

### 7.2 人間チェック運用の3原則

1. **指摘は 1 件ずつ往復せず、まとめて 1 メッセージで渡す。** 会話は毎ターン全履歴を再送するため、10 件を 10 往復で出すと 1 回でまとめて出す場合の数倍の入力トークンを払う。「人間がチェック → 指摘リストを作る → 一括指示」の形にする。
2. **指摘には実値を書く。**「余白がおかしい」では Claude が Figma 再取得や探索を始めてコストが戻ってくる。「#2 ヘッダーの gap を 16px に」「#5 背景を `bg-blue-2` に」まで書けば再取得ゼロで直せる。実値は Figma の Dev Mode で人間が確認する（`mode: report` のレポートには、突き合わせ用にコード上の現在値が併記される）。
3. **粗取りは AI に残す。** 要素の欠落・文言違い・レイアウト崩壊のような「見ればすぐ分かるが数が多いと面倒」な指摘は round 1 でほぼ拾える。`report` モードはこの粗取り 1 回分を含んでいる。完全に検証を省略する（verify を呼ばない）運用は、粗い崩れの見落としが人間チェックの負荷になるため推奨しない。

### 7.3 例外的に `loop` を使ってよい条件

`report` が標準である以上、`loop` を使うには理由が要る。以下のいずれかが成り立つ場合に限り、**ユーザーが明示的に `mode: loop` を指定して**使う:

- **人手が完全に不足しているフェーズ**: 画面数が多く、人間チェックの工数がボトルネックになることが分かっている（AI コスト増 < 人間チェック工数の人件費、が明確に成り立つ）。
- **粗い忠実度で十分と明示されている**: プロトタイプの用途上、細部の品質保証が不要とユーザーが判断している。
- **指摘件数が読めない新規パターンの画面**: `loop` で critical/major を先に潰してから人間が細部を見る方がトータルで安いと見込める場合。この場合も maxRounds=3 の停止条件は必ず維持する。

留意点:

- `report` 標準の隠れコストは、**Figma との突き合わせ作業が人間の工数に移る**こと。上の判断をするときは、画面 1〜2 枚で試行し「`loop` 時の verify 工程コスト（verify 開始前後の `/cost` 差分）」と「`report` + 人間チェックの所要時間」を実測して比較する。
- 例外採用した場合も恒常化させない。フェーズが変わったら標準（`report`）に戻す。

## 8. Agent teams / Dynamic workflows の採否

Claude Code のマルチエージェント機能 **Agent teams**（experimental）と **Dynamic workflows** を本ワークフローに導入すべきか評価した結論（2026-07-05 時点。出典は §8.4）。

### 8.1 結論: どちらも現時点では不採用

本リポジトリの構成（薄いメインループ + 項目単位のサブエージェント委譲・最大3並列バッチ・返答はコンパクト JSON のみ）は、公式の使い分け指針における「結果だけ欲しい少数タスク → subagent（コスト最低）」パターンそのもの。上位機能に乗り換える動機がない。

- `.claude/settings.json` は現状維持（機能有効化の設定を追加しない）
- `.claude/skills/` / `.claude/agents/` の現行設計も変更しない

### 8.2 Agent teams を使わない理由

Agent teams は複数の Claude セッション（lead + teammates）が独立コンテキストで協調し、相互にメッセージ交換する機能。experimental・デフォルト無効で、公式が「significantly more tokens」と明言している。本ワークフローには次の点で不適合:

| 観点 | 不適合の内容 |
|---|---|
| **議論の需要がない** | 強みは teammate 間の相互議論だが、Figma パイプラインは工程が直列（library → flow → implement → verify）。並列化の単位はコンポーネント/画面 1 件で、相互調整は不要 |
| **キャッシュ方針と衝突** | teammate は独立フルセッション（コールドスタート・親のキャッシュ非共有）。§2.3 の「キャッシュヒット率が総額を左右する」方針と真っ向から衝突する |
| **単一書き込み者ルールと競合** | `@theme` はメインループ、`App.tsx` は画面実装エージェントに一本化している（§4.2-5）。並列 teammate の書き込みはこの防壁を壊す |
| **既知の制限** | `/resume` / `/rewind` で in-process teammate が復帰しない、ネストチーム不可、等 |

### 8.3 Dynamic workflows の将来採用条件

Dynamic workflows は JavaScript スクリプトが多数の subagent をオーケストレーションする正式機能。中間結果をスクリプト変数に保持し、Claude のコンテキストには最終レポートのみ戻すため**コスト特性自体は良い**。ただし適用条件は「1 エージェントのコンテキストに収まらない大規模並列」であり、現状のプロトタイプ規模（コンポーネント数十件・画面十数枚）では既存の 3 並列バッチで足りる。

**再検討のしきい値**（いずれかに達したら試行する）:

- コンポーネント 50 件超の一括抽出（`figma-component-extractor` の大規模 fan-out）
- 全画面の verify（`mode: report`）レポートを夜間ランで一括生成する運用が始まったとき

**採用時の設定手順**（リポジトリ側の変更はゼロ・個人設定のみ）:

1. `/config` で「Dynamic workflows」を ON（Pro プラン以上）
2. プロンプトに `ultracode` を含めて起動する（または `/deep-research`）

### 8.4 参考: 3 方式の比較

| 特性 | subagent（現行） | Agent teams | Dynamic workflows |
|---|---|---|---|
| コスト | 最低 | 中〜高（teammate 数に比例） | 高（ただしコンテキストは太らない） |
| コンテキスト | 親に要約のみ戻す | 各 teammate 独立・キャッシュ非共有 | スクリプト変数に中間結果、最終レポートのみ |
| 再開性 | 親セッションの再開で継続 | `/resume` で teammate 消失 | 同一セッション内で再開可 |
| 適用場面 | 結果だけ欲しい少数タスク | 並列探索 + 相互議論が必要 | 計画をコード化できる大規模並列 |
| 有効化 | 不要（標準機能） | `env` に `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS: "1"` | `/config` で ON |

出典: code.claude.com/docs の `agent-teams.md` / `workflows.md` / `costs.md`（Agent team token costs）/ `agents.md`。experimental 機能の仕様・環境変数名は変わりうるため、§1.1 と同じく四半期ごとの見直し対象とする。

## 付録: 運用チェックリスト（日常用）

**タスク開始前**
- [ ] 対象・完了条件・やらないことを 1 プロンプトにまとめたか
- [ ] 前タスクの残骸があるなら `/clear` したか
- [ ] specs に必要情報（node-id・トークン対応）が揃っているか
- [ ] 検証は標準の `report` か。`loop` を使うなら §7.3 の例外条件に該当し、ユーザーが明示指定したか

**人間チェック運用（標準・`mode: report`）のとき**
- [ ] レポートを Figma Dev Mode と突き合わせ、修正指示に実値（px・トークン名・hex）を入れたか
- [ ] 修正指示は 1 件ずつ送らず、まとめて 1 メッセージにしたか

**タスク実行中**
- [ ] verify が 3 round 使い切ったら、続行せず原因（仕様不足？モデル？）を見る
- [ ] 同じエラーが 2 回出たら止める

**タスク終了後**
- [ ] `/cost` を確認し、チーム基準値と比較
- [ ] 想定外に高かった場合、round 数・再スポーン有無を記録して共有
