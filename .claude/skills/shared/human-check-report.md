# 人間チェック用レポートの様式（共通リファレンス）

`figma-verify-screen` / `fix-design-diff` から参照される共通様式。様式A/B は AI レビュー実行時（ユーザー明示依頼時）、様式C は既定フロー（AI レビューなし）の完了案内に使う。AI は指摘・対応結果を整形して渡すまでを担い、取捨選択と修正判断は人間が行う、という分担（`docs/ai-cost-optimization.md` §7）を前提とする。

## 共通原則

- **実値を併記する**: 数値プロパティ（余白・フォント・色）は、コード上の現在値（Tailwind クラス / hex）を必ず書く。人間は Figma の Dev Mode 値と見比べるだけで判断できる。
- **全件載せる**: minor/nit も省略しない（自動修正しないので round を浪費する心配がない。取捨選択は人間の役割）。
- **見比べの材料を揃える**: AI レビューを実行した場合（様式A/B）は実装スクショのパスと参照画像（Figma スクショ / 参照 PNG）のパスを併記する。既定フロー（スクショなし）の場合はブラウザ確認用 URL を案内する（様式C）。見比べ・目視確認は人間が行う。
- 最後に「修正指示の出し方」を1行添える: *指摘はまとめて1メッセージで、実値つきで指示してください（例: 「#2 ヘッダーの gap を 16px に、#5 背景を bg-blue-2 に」）。*（1件ずつの往復は入力トークンを数倍にする — `docs/ai-cost-optimization.md` §7.2）

## 様式A: 検証レポート（figma-verify-screen の findings 用）

| # | severity | 領域 | 指摘 | 該当ファイル:行 | コード上の現在値 | 推定される修正 |
|---|---|---|---|---|---|---|

## 様式B: 相違点対応報告（fix-design-diff の diffList 用）

| # | 相違点リストの項目 | 対応 | 補足（使用トークン・確定 hex 等） |
| - | ------------------ | ---- | --------------------------------- |

表に続けて:

- 変更ファイル一覧（`changedFiles`）
- 未対応項目（`openQuestions`）とその理由
- リスト外で気づいたが手を付けなかった差異（`noticedButSkipped`）— 対応するかは人間が判断

## 様式C: 完了案内（既定フロー・AI レビューなしのとき）

スクショ・AI レビューを経ていない、実装/修正完了時点の軽量な報告。

- 変更ファイル一覧（`changedFiles`）
- 未対応項目（`openQuestions`）とその理由（あれば）
- リスト外で気づいたが手を付けなかった差異（`noticedButSkipped`）（あれば）
- **ブラウザ確認用 URL**: `http://localhost:<port><route>`（port は対象プロトタイプの `vite.config.ts` が権威）
- 参照画像のパス（`fix-design-diff` で `imagePath` を受け取っている場合。人間が見比べる材料として）
- 末尾に1行添える: *見た目の確認はブラウザで行ってください。AI によるスクショ比較・視覚検証が必要な場合は明示的に依頼してください（「Figma と突き合わせて」「見比べて」等）。*
