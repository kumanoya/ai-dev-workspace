実装完了後に、specs/ と実装のズレを解消する。specs はこのリポジトリの Source of Truth であり、その精度が verify の往復回数（= 最大のコスト要因）を直接決める。

## 手順

1. `$ARGUMENTS` から対象の spec ファイルを特定する（`specs/` 配下。画面別・コンポーネント別）。見つからなければ候補を挙げてユーザーに確認する。
2. spec と対応する実装（`prototypes/<name>/src/` 配下）を読み比べ、ドリフトを分類する:
   - **実装が正**（レビュー・修正指示を経て実装側が確定した差分）→ spec を更新
   - **spec が正**（実装漏れ・実装ミスの疑い）→ spec は変えず、指摘として報告
   - **どちらが正か不明** → 変更せず、判断材料を添えて報告
3. spec の更新はドキュメント方針（CLAUDE.md）に従う: 実装コードの丸写しをせず、構造・値・制約を仕様として記述する。「以前は◯◯だった」等の経緯注記は書かない。
4. 更新した spec・報告事項（実装ミスの疑い・判断保留）を一覧で報告する。

## 制約

- 実装コード（.tsx 等）は修正しない。spec の更新と指摘のみ。
- `specs/spec.md`（グローバル仕様）に影響する変更は勝手に反映せず、提案としてユーザーに確認する。
